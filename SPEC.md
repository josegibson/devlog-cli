# devlog — agent bootstrap specification

This document is the source of truth for building `devlog`.
Read it fully before writing any code. It contains the architecture,
schema, command surface, generation rules, and open decisions.

---

## what devlog is

A **project state engine** whose primary user is the coding agent, not the developer.
The developer expresses intent. The agent logs everything else.
`AGENTS.md` is the live projection of the store — auto-regenerated on every write.
Every coding agent reads `AGENTS.md` on session start as its onboarding contract.

---

## core invariant

```
.devlog/ YAML store  →  CLI regenerates  →  AGENTS.md  →  agent reads at session start
        ↑                                                          ↓
agent writes (call / snag / shift / observe)              agent operates
```

The CLI owns `AGENTS.md`. The agent never writes it directly.
The agent writes records. The CLI projects them.

---

## memory architecture

### tier 1 — working memory
- File: `.devlog/scratch.yaml`
- Content: raw agent `observe` logs from the current and recent sessions
- Lifespan: ephemeral. Archived (not deleted) when compression triggers.
- Never injected into `AGENTS.md` in full — only the last N entries appear.

### tier 2 — long-term memory
- Files: `.devlog/calls.yaml`, `snags.yaml`, `shifts.yaml`, `intents.yaml`,
  `milestones.yaml`, `constraints.yaml`, `briefs.yaml`
- Content: structured, typed records written by agent or developer
- Lifespan: permanent. Superseded records stay with updated status.
- This is what `AGENTS.md` is projected from.

### tension map — live derived state
- File: `.devlog/tension.yaml` (auto-derived, never hand-edited)
- Content: every active `call` with its current `confidence` field,
  linked open `snags`, and pending `needs-review` flags
- Regenerated by the CLI on every write to calls/snags/shifts
- Always included in `AGENTS.md` in full

## command surface

### human commands (developer writes these — 4 only)

```bash
devlog intent "ship passenger counting pipeline to i.MX 93" \
  --why "IBVAS phase 2 delivery" \
  --done "model runs on-device at >85% accuracy with zone counting"

devlog milestone "v0.1 — on-device inference working" \
  --version "0.1.0"

devlog constraint "no YOLOv8 — zero NPU nodes on Ethos-U65" \
  --source "architecture decision confirmed with Jay"

devlog inspect
# renders tension map + active intent to stdout in plain language
```

`inspect` is read-only. It never writes to the store.
`constraint` records are immutable — removing one requires `devlog constraint remove <id>`.

---

### agent commands (agent writes these autonomously)

```bash
# Decision made
devlog call "use MobileNetV2+SSD for detection backbone" \
  --context "NPU compatibility required, YOLOv8 ruled out by constraint-001" \
  --over "EfficientDet, YOLOv5" \
  --to-achieve "full NPU delegation on Ethos-U65" \
  --tradeoff "lower mAP ceiling than YOLOv8 on overhead views"

# Blocker hit — --threatens is NOT optional for agent
devlog snag "overhead pedestrian domain gap — pretrained weights fail on top-down views" \
  --threatens "call-2026-05-13-mobilenetv2-ssd" \
  --impact high

# Assumption broke — triggers tension engine
devlog shift \
  --from "PAMELA-UANDES dataset for overhead training data" \
  --to "Gemini API auto-labeling pipeline on custom frames" \
  --assumption-broke "public overhead pedestrian datasets are accessible" \
  --sustained "MobileNetV2+SSD backbone still valid"

# Raw fact — goes to tier 1 scratch only
devlog observe "SSDLite iteration 4 — mAP 0.61 on validation set"
devlog observe "RunPod A4000 — 3.2hr training run, cost $4.80"

# Compression — agent writes this at milestone or major shift
devlog brief \
  --situation "v0.1 inference pipeline running on i.MX 93 dev board" \
  --background "switched from public datasets to Gemini auto-labeling after PAMELA blocked" \
  --assessment "accuracy acceptable for phase 2, zone counting not yet implemented" \
  --recommendation "implement ByteTrack + zone boundary logic before accuracy tuning"
```

## tension engine rules

Runs automatically on every write to `calls.yaml`, `snags.yaml`, or `shifts.yaml`.
Implemented as a CLI side-effect — not a separate daemon.

```
on shift filed:
  extract assumption-broke value
  scan all calls where context or to-achieve semantically overlaps
  mark matched calls → confidence: degraded

on snag filed with --threatens:
  find referenced call by id
  mark that call → confidence: at-risk

on any call not referenced (by snag, shift, or milestone) for > SESSION_STALE_THRESHOLD:
  mark call → confidence: unconfirmed

on milestone filed:
  all open calls with confidence: nominal → confidence: confirmed-at-milestone
  trigger compression
```

`SESSION_STALE_THRESHOLD` defaults to 5 sessions. Configurable in `.devlog/config.yaml`.

Confidence states: `nominal` | `degraded` | `at-risk` | `unconfirmed` | `confirmed-at-milestone`

---

## compression trigger

Fires on: `devlog milestone` filed, `devlog shift` of impact `high`, or
`scratch.yaml` entry count > `COMPRESS_THRESHOLD` (default: 50).

On compression:
1. Agent writes a `devlog brief` summarizing the tier 1 window
2. CLI moves processed `scratch.yaml` entries to `.devlog/archive/scratch-<timestamp>.yaml`
3. CLI regenerates `AGENTS.md` from current tier 2 state

## record schemas

### intent
```yaml
id: intent-2026-05-13-ibvas-pipeline
timestamp: 2026-05-13T10:00:00Z
author: human
goal: "ship passenger counting pipeline to i.MX 93"
why: "IBVAS phase 2 delivery"
done_looks_like: "model runs on-device at >85% accuracy with zone counting"
status: active  # active | superseded | achieved
```

### call
```yaml
id: call-2026-05-13-mobilenetv2-ssd
timestamp: 2026-05-13T10:45:00Z
author: agent
decision: "use MobileNetV2+SSD for detection backbone"
context: "NPU compatibility required, YOLOv8 ruled out by constraint-001"
rejected_alternatives:
  - EfficientDet
  - YOLOv5
to_achieve: "full NPU delegation on Ethos-U65"
tradeoff: "lower mAP ceiling than YOLOv8 on overhead views"
confidence: nominal  # nominal | degraded | at-risk | unconfirmed | confirmed-at-milestone
status: active  # active | superseded
superseded_by: null
intent_ref: intent-2026-05-13-ibvas-pipeline
```

### snag
```yaml
id: snag-2026-05-14-overhead-domain-gap
timestamp: 2026-05-14T09:20:00Z
author: agent
description: "overhead pedestrian domain gap — pretrained weights fail on top-down views"
threatens: call-2026-05-13-mobilenetv2-ssd
blocks: "training run iteration 1"
impact: high  # low | medium | high
status: open  # open | cleared
cleared_by: null
```

### shift
```yaml
id: shift-2026-05-15-dataset-pivot
timestamp: 2026-05-15T14:10:00Z
author: agent
from: "PAMELA-UANDES dataset for overhead training data"
to: "Gemini API auto-labeling pipeline on custom frames"
assumption_broke: "public overhead pedestrian datasets are accessible"
sustained: "MobileNetV2+SSD backbone still valid"
intent_ref: intent-2026-05-13-ibvas-pipeline
```

### observe (tier 1 only)
```yaml
- timestamp: 2026-05-15T16:00:00Z
  text: "SSDLite iteration 4 — mAP 0.61 on validation set"
- timestamp: 2026-05-15T19:30:00Z
  text: "RunPod A4000 — 3.2hr training run, cost $4.80"
```

### constraint
```yaml
id: constraint-001
timestamp: 2026-05-13T09:00:00Z
author: human
constraint: "no YOLOv8 — zero NPU nodes on Ethos-U65"
source: "architecture decision confirmed with Jay"
immutable: true
```

### milestone
```yaml
id: milestone-2026-05-20-v0-1
timestamp: 2026-05-20T18:00:00Z
author: human
title: "v0.1 — on-device inference working"
version: "0.1.0"
achieved: null  # set by agent when filing brief at this milestone
calls_confirmed: []  # populated by tension engine on milestone file
```

### brief
```yaml

timestamp: 2026-05-20T18:05:00Z
author: agent
milestone_ref: milestone-2026-05-20-v0-1
situation: "v0.1 inference pipeline running on i.MX 93 dev board"
background: "switched from public datasets to Gemini auto-labeling after PAMELA blocked"
assessment: "accuracy acceptable for phase 2, zone counting not yet implemented"
recommendation: "implement ByteTrack + zone boundary logic before accuracy tuning"
```

## AGENTS.md template

Generated deterministically by CLI from tier 2 state.
Never edited by hand. Committed to git on every regeneration.

```markdown
# project state — <project-name>
> generated by devlog · <timestamp> · do not edit manually

## active intent
**goal:** <intent.goal>
**why:** <intent.why>
**done looks like:** <intent.done_looks_like>

## constraints (immutable)
- [constraint-001] <constraint.constraint> — <constraint.source>

## tension map
### decisions
| id | decision | confidence | tradeoff |
|----|----------|------------|----------|
| call-... | <decision> | nominal | <tradeoff> |
| call-... | <decision> | at-risk ⚠️ | <tradeoff> |

### open snags
| id | description | threatens | impact |
|----|-------------|-----------|--------|
| snag-... | <description> | call-... | high |

### needs review
- call-... marked degraded by shift-... — assumption broke: "<assumption_broke>"

## current milestone
**target:** <milestone.title> (<milestone.version>)
**achieved:** <null or timestamp>

## last brief
**situation:** <brief.situation>
**assessment:** <brief.assessment>
**recommendation:** <brief.recommendation>

## recent observations (last 10)
- <timestamp> <observe.text>
- ...
```

## storage layout

```
.devlog/
  config.yaml          # SESSION_STALE_THRESHOLD, COMPRESS_THRESHOLD
  intents.yaml
  calls.yaml
  snags.yaml
  shifts.yaml
  constraints.yaml
  milestones.yaml
  briefs.yaml
  tension.yaml         # derived, auto-regenerated
  scratch.yaml         # tier 1 working memory
  archive/
    scratch-<ts>.yaml  # compressed tier 1 segments

AGENTS.md              # generated projection — committed to git
```

All files are append-only YAML lists except `tension.yaml` (fully rewritten on each update).
Git history is the audit trail. No database.

## CLI implementation notes

- Language: Python with `uv` (consistent with Jose's toolchain on mbair)
- Entry point: `devlog <command> [args] [flags]`
- On every write command: update the relevant YAML → run tension engine → regenerate `AGENTS.md`
- `devlog init` creates the `.devlog/` directory and writes an empty `AGENTS.md`
- IDs are deterministic: `<type>-<YYYY-MM-DD>-<slugified-first-5-words>`
- All timestamps UTC ISO 8601

## open decisions for the agent to resolve

These are not specified. Agent should file a `devlog call` for each one before implementing.

1. **Semantic overlap detection in tension engine** — how does the CLI determine that a
   shift's `assumption_broke` overlaps with a call's `context`? Options: keyword match,
   embeddings, explicit `--domain` tag on calls. Each has cost/accuracy tradeoffs.

2. **`devlog inspect` output format** — stdout plain text vs. rendered markdown vs.
   opening a browser GUI. Relevant to the GUI roadmap.

3. **Session boundary definition** — what constitutes a "session" for
   `SESSION_STALE_THRESHOLD`? Git commit? devlog command invocation? Explicit
   `devlog session start/end`? Affects staleness detection reliability.

4. **`brief` authorship** — should the agent always write the brief manually, or should
   the CLI auto-generate a brief from recent tier 1 entries on compression trigger,
   with the agent reviewing? Auto-generation is lower friction but lower quality.

---

## success criteria

The system is working if:
- A new agent session can read `AGENTS.md` and know exactly what is decided,
  what is uncertain, what is blocked, and what to do next
- The developer never has to explain project history to the agent
- Stale decisions surface before they silently corrupt new work
- The developer's only required input is intent, milestones, and constraints
- The full decision history is recoverable from git at any point
